## 2. 컴포넌트별 역할

### 2.1 Frontend (Vue.js)

| 컴포넌트 | 역할 |
|---------|------|
| LoginView.vue | 로그인·회원가입 탭 전환 UI |
| MainView.vue | 법령 필터, 질문 입력, 답변·출처 표시 |
| HistoryView.vue | 회원 전용 질의응답 히스토리 조회 |
| stores/auth.js | Pinia — JWT 토큰·사용자 상태 관리 |
| router/index.js | Vue Router — 비회원/회원 접근 제어 |

### 2.2 Backend (FastAPI)

| 모듈 | 역할 |
|------|------|
| main.py | FastAPI 앱 진입점, API 라우터 등록 |
| auth.py | 회원가입·로그인·JWT 발급 API |
| rag_pipeline.py | RAG 파이프라인 (임베딩·검색·답변 생성) |
| law_loader.py | 국가법령정보센터 API 수집·조문 파싱·ChromaDB 적재 |
| models.py | SQLAlchemy ORM 모델 (User, QAHistory) |
| database.py | PostgreSQL 연결 설정 |
| jwt_utils.py | JWT 토큰 생성·검증 유틸리티 |

### 2.3 데이터 저장소

| 저장소 | 용도 | 저장 데이터 |
|--------|------|------------|
| PostgreSQL | 관계형 데이터 | 회원 정보, 질의응답 히스토리 |
| ChromaDB | 벡터 데이터 | 법령 조문 임베딩 벡터 + 메타데이터 |

## 3. RAG 파이프라인 상세 흐름

### 3.1 법령 데이터 적재 흐름 (최초 1회 / 관리자)

```
국가법령정보센터 Open API
          │
          │ XML 응답 수신
          ▼
    law_loader.py
          │
          │ 조문 단위 파싱
          │ (제1조, 제2조 ... 항·호 단위)
          ▼
    메타데이터 태깅
    {
      "law_name": "근로기준법",
      "article_no": "제60조",
      "article_title": "연차 유급휴가",
      "content": "..."
    }
          │
          │ ko-sroberta-multitask 임베딩
          ▼
      ChromaDB 저장
    (collection: "law_articles")
```

### 3.2 질의응답 흐름

```
사용자 질문 입력
"연차는 며칠 쓸 수 있나요?"
          │
          ▼
  ko-sroberta 임베딩
  질문 → 벡터 변환
          │
          ▼
  ChromaDB 유사도 검색
  Top-K 관련 조문 반환 (K=4)
          │
          ▼
  LangChain 프롬프트 구성
  [시스템 지시 + 조문 컨텍스트 + 질문]
          │
          ▼
  GPT-4o-mini API 호출
          │
          ▼
  답변 + 출처(법령명·조문번호) 반환
          │
          ├─ 비회원: 세션 내 표시만
          └─ 회원:   PostgreSQL qa_history 저장
```

## 4. API 구조 개요

### 4.1 인증 API

| Method | Endpoint | 설명 | 권한 |
|--------|----------|------|------|
| POST | /api/v1/auth/signup | 회원가입 | 비회원 |
| POST | /api/v1/auth/login | 로그인 (JWT 발급) | 비회원 |

### 4.2 질의응답 API

| Method | Endpoint | 설명 | 권한 |
|--------|----------|------|------|
| POST | /api/v1/qa/ask | 질문 → 답변 생성 | 비회원·회원 |
| GET | /api/v1/qa/history | 히스토리 조회 | 회원 전용 |

### 4.3 법령 관리 API

| Method | Endpoint | 설명 | 권한 |
|--------|----------|------|------|
| POST | /api/v1/admin/laws/load | 법령 데이터 적재 | 관리자 |
| GET | /api/v1/admin/laws | 적재된 법령 목록 조회 | 관리자 |

## 5. DB 설계 개요

### 5.1 users 테이블

| 컬럼 | 타입 | 설명 |
|------|------|------|
| id | INTEGER PK | 사용자 ID |
| username | VARCHAR(50) | 사용자명 (unique) |
| email | VARCHAR(100) | 이메일 (unique) |
| password_hash | VARCHAR(255) | bcrypt 해시 비밀번호 |
| role | VARCHAR(20) | USER / ADMIN |
| created_at | DATETIME | 가입일시 |

### 5.2 qa_history 테이블

| 컬럼 | 타입 | 설명 |
|------|------|------|
| id | INTEGER PK | 히스토리 ID |
| user_id | INTEGER FK | users.id 참조 |
| question | TEXT | 사용자 질문 |
| answer | TEXT | LLM 생성 답변 |
| sources | TEXT | 출처 조문 JSON |
| created_at | DATETIME | 질문일시 |

### 5.3 ChromaDB 컬렉션 (law_articles)

| 필드 | 설명 |
|------|------|
| id | 조문 고유 ID |
| embedding | ko-sroberta 임베딩 벡터 |
| document | 조문 원문 텍스트 |
| metadata.law_name | 법령명 |
| metadata.article_no | 조문 번호 |
| metadata.article_title | 조문 제목 |

## 7. 보안 설계

| 항목 | 내용 |
|------|------|
| 인증 방식 | JWT Bearer Token (Access: 24시간) |
| 비밀번호 | bcrypt 해시 저장 |
| CORS | 개발 환경 전체 허용, 운영 시 도메인 제한 |
| 비회원 QA | 인증 없이 질문 가능 (rate limit 고려) |
| SECRET_KEY | .env 파일로 관리, Git 미포함 |
