# 📋 RAG 파이프라인 설계서

> **프로젝트명:** 법령 RAG 문서 검색 시스템 (Legal RAG Document Search System)
> **작성일:** 2026년 6월
> **작성자:** 전보경
> **버전:** v1.0

---

## 1. 개요

본 문서는 법령 RAG 문서 검색 시스템의 AI 파이프라인 설계를 상세히 기술한다.
크게 두 가지 파이프라인으로 구성된다.

- **적재 파이프라인:** 법령 원문 수집 → 조문 파싱 → 임베딩 → ChromaDB 저장
- **질의응답 파이프라인:** 질문 임베딩 → 유사도 검색 → 프롬프트 구성 → LLM 답변 생성

---

## 2. 적재 파이프라인 (Indexing Pipeline)

### 2.1 전체 흐름

```
국가법령정보센터 Open API
        │
        │ GET /DRF/lawService.do?target=law&ID={법령ID}&type=XML
        ▼
  XML 원문 수신
        │
        ▼
  조문 단위 파싱 (law_loader.py)
        │
        ▼
  메타데이터 태깅
        │
        ▼
  청크 길이 검증 및 분할
        │
        ▼
  ko-sroberta-multitask 임베딩
        │
        ▼
  ChromaDB 저장 (collection: law_articles)
```

### 2.2 적재 대상 법령

| 법령명 | 법령 ID | 주요 조문 수 (약) |
|--------|---------|-----------------|
| 근로기준법 | 293 | 약 110개 |
| 최저임금법 | 360 | 약 30개 |
| 근로자퇴직급여 보장법 | 285 | 약 50개 |
| 남녀고용평등법 | 287 | 약 40개 |
| 고용보험법 | 302 | 약 80개 |

### 2.3 조문 파싱 규칙

국가법령정보센터 API의 XML 응답 구조를 기반으로 조문을 파싱한다.

```xml
<!-- API 응답 XML 구조 예시 -->
<법령>
  <조문>
    <조문번호>제60조</조문번호>
    <조문제목>연차 유급휴가</조문제목>
    <조문내용>
      ① 사용자는 1년간 80퍼센트 이상 출근한 근로자에게...
      ② 사용자는 계속하여 근로한 기간이 1년 미만인 근로자...
    </조문내용>
  </조문>
</법령>
```

**파싱 단위 결정: 조문(Article) 단위**

항(①②③) 단위로 더 쪼개는 방법도 있지만 조문 단위를 선택한 이유는 다음과 같다.

- 항은 단독으로 의미가 완결되지 않는 경우가 많음
- 조문 단위가 법령의 의미 단위로 가장 자연스러움
- 검색 시 관련 항이 함께 반환되어 컨텍스트 손실 방지

### 2.4 메타데이터 설계

각 청크에 다음 메타데이터를 태깅한다.

```python
{
    "law_name":      "근로기준법",          # 법령명
    "article_no":    "제60조",              # 조문 번호
    "article_title": "연차 유급휴가",        # 조문 제목
    "law_id":        "293",                 # 법령 ID
    "collected_at":  "2026-06-14"           # 수집일
}
```

### 2.5 청크 길이 처리

| 조문 길이 | 처리 방식 |
|----------|----------|
| 1,000자 이하 | 조문 전체를 단일 청크로 사용 |
| 1,000자 초과 | 항(①②③) 단위로 분할하여 복수 청크 생성 |

### 2.6 임베딩 모델

| 항목 | 내용 |
|------|------|
| 모델 | jhgan/ko-sroberta-multitask |
| 유형 | Sentence Transformer (한국어 특화) |
| 출력 차원 | 768차원 벡터 |
| 실행 환경 | 로컬 (CUDA — RTX 3050) |
| 선택 이유 | 한국어 법률 용어 간 의미 유사도 정확도 우수, 무료 |

```python
from langchain_community.embeddings import HuggingFaceEmbeddings

embed_model = HuggingFaceEmbeddings(
    model_name="jhgan/ko-sroberta-multitask",
    model_kwargs={"device": "cuda"},
    encode_kwargs={"normalize_embeddings": True}
)
```

### 2.7 중복 적재 방지

법령 ID + 조문 번호 조합을 고유 ID로 사용하여 ChromaDB에 이미 존재하는 청크는 재적재하지 않는다.

```python
chunk_id = f"{law_id}_{article_no}"  # 예: "293_제60조"
```

## 3. 질의응답 파이프라인 (Query Pipeline)

### 3.1 전체 흐름

```
사용자 질문
"연차는 며칠 쓸 수 있나요?"
        │
        ▼
  법령 필터 적용 (선택)
  예: 근로기준법만 검색
        │
        ▼
  질문 임베딩
  ko-sroberta → 768차원 벡터
        │
        ▼
  ChromaDB 유사도 검색
  코사인 유사도 기반 Top-4 조문 반환
        │
        ▼
  유사도 임계값 필터 (threshold: 0.3)
  관련 조문 없으면 "찾을 수 없음" 반환
        │
        ▼
  LangChain 프롬프트 구성
  [시스템 지시 + 조문 컨텍스트 + 질문]
        │
        ▼
  GPT-4o-mini API 호출
  temperature=0 (일관된 답변)
        │
        ▼
  답변 + 출처 반환
  {
    "answer": "...",
    "sources": [
      {"law_name": "근로기준법", "article_no": "제60조", "content": "..."}
    ]
  }
        │
        ├─ 비회원: 세션 내 표시
        └─ 회원:   PostgreSQL 저장 후 표시
```

### 3.2 검색 전략

| 항목 | 설정값 | 이유 |
|------|--------|------|
| 검색 방식 | 코사인 유사도 | 텍스트 의미 유사도 측정에 적합 |
| Top-K | 4 | 컨텍스트 과다 시 LLM 성능 저하, 4개가 적정 |
| 유사도 임계값 | 0.3 | 관련 없는 조문 반환 방지 |
| 법령 필터 | metadata.law_name | 특정 법령만 검색 시 필터 적용 |

### 3.3 프롬프트 설계

```python
SYSTEM_PROMPT = """당신은 대한민국 노동법령 전문 안내 도우미입니다.
반드시 아래 규칙을 따르세요.

1. 제공된 법령 조문 내용만을 근거로 답변하세요.
2. 조문에 없는 내용은 "제공된 법령에서 해당 정보를 찾을 수 없습니다."라고 답변하세요.
3. 답변은 누구나 이해하기 쉬운 언어로 작성하세요.
4. 답변 마지막에 반드시 면책 문구를 포함하세요:
   "※ 본 답변은 참고용이며 법적 효력이 있는 해석이 아닙니다."
"""

RAG_PROMPT = PromptTemplate(
    input_variables=["context", "question"],
    template="""[참고 법령 조문]
{context}

[질문]
{question}

[답변]"""
)
```

### 3.4 컨텍스트 구성 방식

검색된 Top-4 조문을 다음 형식으로 구성하여 LLM에 전달한다.

```
[근로기준법 제60조 (연차 유급휴가)]
사용자는 1년간 80퍼센트 이상 출근한 근로자에게 15일의 유급휴가를 주어야 한다...

[근로기준법 제61조 (연차 유급휴가의 사용 촉진)]
사용자가 제60조에 따른 유급휴가의 사용을 촉진하기 위하여...
```

## 5. 성능 고려사항

| 항목 | 내용 |
|------|------|
| 임베딩 속도 | RTX 3050 CUDA 활용으로 배치 임베딩 가속 |
| 최초 적재 시간 | 법령 5개 기준 약 5~10분 예상 |
| 질의응답 응답 시간 | 임베딩(~0.1초) + 검색(~0.1초) + LLM(~3~5초) = 약 5초 이내 목표 |
| ChromaDB 영구 저장 | Docker volume 마운트로 컨테이너 재시작 시에도 데이터 유지 |
